## 2. ENTITIES (Database Tables)

Spring Boot creates these as **actual tables** in MySQL automatically.

### `User.java`

```java
@Entity                     // "This becomes a database table"
@Table(name = "users")      // Table name will be "users"
public class User {

    @Id                     // Primary key
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;        // Auto-increment number (1, 2, 3...)

    private String name;        // User's full name (required)
    private String email;       // Unique email (required, used for login)
    private String password;    // Encrypted password (required)
    private String phoneNumber; // Optional phone
    private String company;     // Optional company name
    private Integer age;        // Age (must be 13-120)
    private Gender gender;      // MALE, FEMALE, OTHER, PREFER_NOT_TO_SAY
    private Role role;          // ROLE_USER or ROLE_ADMIN
    private LocalDateTime createdAt;  // Auto-set when account is created

    // One user can have MANY documents
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
    private List<Document> documents = new ArrayList<>();

    @PrePersist
    void onCreate() {
        if (role == null) role = Role.ROLE_USER;  // Default role
        createdAt = LocalDateTime.now();          // Set creation time
    }
}
```

**Plain English:** The `User` table stores every registered account. Each user has an ID, name, email (used to login), encrypted password, and optional profile info. The `@OneToMany` line means "one user can own many documents" — if you delete a user, all their documents get deleted too (cascade).

### `Document.java`

```java
@Entity
@Table(name = "documents")
public class Document {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne                          // Many documents belong to ONE user
    @JoinColumn(name = "user_id")
    private User user;

    private String documentName;        // e.g., "PUC Certificate"
    private String documentType;        // e.g., "Insurance"
    private String permitNumber;        // e.g., "MH03-ABC-1234"
    private LocalDate issueDate;        // When was it issued?
    private LocalDate expiryDate;       // When does it expire?
    private String authorityName;       // Which authority issued it?

    private String originalFileName;    // Original name (e.g., "puc.pdf")
    private String storedFileName;      // Unique name on disk (UUID.pdf)
    private String filePath;            // Full path to the stored file

    @Lob                                // Large text field
    @Column(columnDefinition = "LONGTEXT")
    private String extractedText;       // Raw text from AI after upload

    private LocalDateTime uploadTime;   // When was it uploaded?
    private DocumentStatus status;      // ACTIVE, EXPIRING_SOON, or EXPIRED
}
```

**Plain English:** Each row in `documents` represents one uploaded permit/certificate. It stores the extracted data (like expiry date) from Gemini AI, the file path on disk, and a status that changes automatically as the expiry date approaches.

## 3. DTOs (Data Transfer Objects)

DTOs carry data between the browser and the server. They are NOT saved to the database — they're temporary.

### `SignupRequest.java`

```java
public class SignupRequest {
    @NotBlank @Size(max = 120)        // Cannot be empty, max 120 chars
    private String name;

    @NotBlank @Email @Size(max = 160) // Must be a valid email format
    private String email;

    @NotBlank @Size(min = 8, max = 100) // Must be at least 8 characters
    private String password;

    private String phoneNumber;       // Optional
    private String company;           // Optional

    @NotNull @Min(13) @Max(120)       // Age 13-120
```

**Plain English:** When someone fills the signup form, the browser sends this data to our server. The `@NotBlank`, `@Email`, etc. are **validation annotations** — Spring checks them automatically. If the user puts an invalid email or leaves a required field empty, Spring rejects the request.

## 4. REPOSITORIES (Database Access)

Repositories are the bridge between Java code and MySQL. Spring Data JPA generates the SQL queries automatically.

### `UserRepository.java`

```java
public interface UserRepository extends JpaRepository<User, Long> {

    Optional<User> findByEmailIgnoreCase(String email);
    // SQL: SELECT * FROM users WHERE LOWER(email) = ?

    boolean existsByEmailIgnoreCase(String email);
    // SQL: SELECT COUNT(*) > 0 FROM users WHERE LOWER(email) = ?
}
```

**Plain English:** `JpaRepository<User, Long>` means "this manages User objects with Long IDs". Spring auto-generates `save()`, `findById()`, `findAll()`, `delete()`, `count()`. We add two custom methods: one to find a user by email (for login) and one to check if an email already exists (for signup).

### `DocumentRepository.java`

```java
public interface DocumentRepository extends JpaRepository<Document, Long> {

    // Get ALL documents for a user, sorted newest first
    Page<Document> findByUserOrderByUploadTimeDesc(User user, Pageable pageable);
    List<Document> findByUserOrderByUploadTimeDesc(User user);

    // Get ONE specific document owned by a specific user
    Optional<Document> findByIdAndUser(Long id, User user);

    // Search documents by name (case-insensitive)
    Page<Document> findByUserAndDocumentNameContainingIgnoreCaseOrderByUploadTimeDesc(
        User user, String documentName, Pageable pageable
    );

    // Find documents expiring between two dates
    List<Document> findByUserAndExpiryDateBetweenOrderByExpiryDateAsc(
        User user, LocalDate start, LocalDate end
    );

    // Find documents with a specific status and expiry date
    List<Document> findByStatusAndExpiryDate(DocumentStatus status, LocalDate expiryDate);
}
```

**Plain English:** Method names tell Spring what query to build:
- `findByUserOrderByUploadTimeDesc` → "WHERE user = ? ORDER BY upload_time DESC"
- `findByIdAndUser` → "WHERE id = ? AND user = ?" (security: only get YOUR documents)
- `findByUserAndExpiryDateBetween` → "WHERE user = ? AND expiry_date BETWEEN ? AND ?"
- `Page` returns results in pages (e.g., 10 per page)

## 6. SECURITY (Login, Passwords, JWT)

### `SecurityConfig.java`

```java
@Configuration
@EnableMethodSecurity
public class SecurityConfig {

    @Bean
    SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(AbstractHttpConfigurer::disable)    // Disable CSRF for APIs
            .authorizeHttpRequests(auth -> auth
                // These URLs are PUBLIC (no login needed):
                .requestMatchers("/", "/login", "/signup", "/error",
                    "/api/auth/**", "/api/iot/**",
                    "/v3/api-docs/**", "/swagger-ui/**",
                    "/css/**", "/js/**").permitAll()

                // These URLs need ADMIN role:
                .requestMatchers("/admin/**").hasRole("ADMIN")

                // EVERYTHING ELSE needs login:
                .anyRequest().authenticated()
            )
            // Login form settings:
            .formLogin(form -> form
                .loginPage("/login")
                .usernameParameter("email")      // Use email as username
                .defaultSuccessUrl("/dashboard", true)
                .failureUrl("/login?error=true")
            )
            // Logout settings:
            .logout(logout -> logout
                .logoutUrl("/logout")
                .logoutSuccessUrl("/login?logout=true")
                .invalidateHttpSession(true)     // Clear session
                .deleteCookies("JSESSIONID")     // Remove cookie
            )
            .addFilterBefore(jwtAuthenticationFilter, ...);

        return http.build();
    }
}
```

**Plain English:** This is the security guard. It decides:
- `/login`, `/signup` → anyone can access
- `/admin/**` → only ROLE_ADMIN users
- Everything else → must be logged in
- Login form uses email as username (not the default "username" field)
- On logout, session is destroyed

### `UserPrincipal.java`

```java
public record UserPrincipal(
    Long id, String name, String email, String password, Role role
) implements UserDetails {

    public static UserPrincipal from(User user) {
        return new UserPrincipal(
            user.getId(), user.getName(), user.getEmail(),
            user.getPassword(), user.getRole()
        );
    }

    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        return List.of(new SimpleGrantedAuthority(role.name()));
        // Converts ROLE_ADMIN → "ROLE_ADMIN" authority
    }

    @Override
    public String getUsername() { return email; }

    @Override
    public String getPassword() { return password; }
}
```

**Plain English:** This wraps our `User` entity to match Spring Security's expectations. The three key things Spring needs:
1. `getUsername()` → we return email (since we login with email)
2. `getPassword()` → the encrypted password
3. `getAuthorities()` → the user's role (controls what they can access)

### `JwtTokenProvider.java`

```java
@Component
public class JwtTokenProvider {

    public String generateToken(UserPrincipal principal) {
        return Jwts.builder()
            .subject(principal.email())         // Who is this token for?
            .claim("userId", principal.id())     // Extra data
            .claim("role", principal.role().name())
            .issuedAt(new Date())
            .expiration(new Date(now + expirationMillis))  // 24 hours from now
            .signWith(secretKey)                 // Sign with our secret
            .compact();                          // Create the token string
    }

    public String getUsername(String token) {
        return parseClaims(token).getSubject();  // Extract email from token
    }

    public boolean validateToken(String token) {
        parseClaims(token);  // Throws if invalid/expired
        return true;
    }
}
```

**Plain English:** JWT = JSON Web Token. After login, we create a token that contains the user's email, ID, and role. This token is sent back to the browser. For every future request, the browser sends this token in the header. We validate it to know who's making the request.

### `JwtAuthenticationFilter.java`

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, ...) {
        String token = resolveToken(request);  // Extract "Bearer xxx" from header

        if (token != null) {
            jwtTokenProvider.validateToken(token);
            String email = jwtTokenProvider.getUsername(token);
            var userDetails = userDetailsService.loadUserByUsername(email);

            // Tell Spring: "this user is logged in"
            SecurityContextHolder.getContext().setAuthentication(
                new UsernamePasswordAuthenticationToken(userDetails, null, ...)
            );
        }
        filterChain.doFilter(request, response);  // Continue the request
    }

    private String resolveToken(HttpServletRequest request) {
        String header = request.getHeader("Authorization");
        if (header != null && header.startsWith("Bearer ")) {
            return header.substring(7);  // Remove "Bearer " prefix
        }
        return null;
    }
}
```

**Plain English:** This runs on EVERY request. It checks the `Authorization` header: if it contains `Bearer <token>`, it validates the token and logs the user in automatically. This is how REST API calls stay authenticated without session cookies.

```java
        // Create uploads/ folder if needed

        // Generate unique filename: e.g., "a1b2c3d4-e5f6-7890.png"
        String extension = getExtension(file.getOriginalFilename());
        String storedName = UUID.randomUUID() + "." + extension;

        // Save file to uploads/ folder
        Path target = uploadRoot.resolve(storedName);
        Files.copy(file.getInputStream(), target, REPLACE_EXISTING);

        return new StoredFile(originalName, storedName, target.toString(), contentType);
    }

    public void delete(String filePath) {
        Files.deleteIfExists(Path.of(filePath));
    }

    private void validate(MultipartFile file) {
        if (file.isEmpty()) throw new FileStorageException("File is required");
        if (file.getSize() > maxFileSizeBytes) throw new FileStorageException("File too large");

        String ext = getExtension(file.getOriginalFilename());
        if (!ALLOWED_EXTENSIONS.contains(ext) || !ALLOWED_TYPES.contains(file.getContentType()))
            throw new FileStorageException("Only PDF, JPG, PNG allowed");
    }
```

### `CountdownService.java` → `CountdownServiceImpl.java`

```java
@Service
public class CountdownServiceImpl implements CountdownService {

    public CountdownResponse calculate(LocalDate expiryDate) {
        if (expiryDate == null) return zeroCountdown();

        // Calculate: how long from NOW until expiry date midnight?
        LocalDateTime expiryEnd = expiryDate.plusDays(1).atStartOfDay();
        Duration duration = Duration.between(LocalDateTime.now(), expiryEnd);

        if (duration.isNegative() || duration.isZero()) {
            return expired();
        }

        return CountdownResponse.builder()
            .remainingDays(duration.toDays())        // e.g., 25 days
            .remainingHours(duration.toHoursPart())  // e.g., 6 hours
            .remainingMinutes(duration.toMinutesPart()) // e.g., 23 minutes
            .expired(false)
            .build();
    }

    public DocumentStatus statusFor(LocalDate expiryDate) {
        if (expiryDate == null) return DocumentStatus.ACTIVE;
        LocalDate today = LocalDate.now();

        if (expiryDate.isBefore(today)) return DocumentStatus.EXPIRED;
        if (!expiryDate.isAfter(today.plusDays(expiringSoonDays)))
            return DocumentStatus.EXPIRING_SOON;
        return DocumentStatus.ACTIVE;
    }
}
```

**Plain English:** The countdown timer that shows on the dashboard. `calculate()` returns days/hours/minutes remaining. `statusFor()` decides the status:
- Past expiry → EXPIRED
- Within 30 days → EXPIRING_SOON (yellow warning)
- More than 30 days → ACTIVE (green)

## 11. How Data Flows Through The App

### Registration Flow:
```
Browser fills signup form
    ↓
POST /signup with form data
    ↓
AuthViewController handles it
    ↓
AuthService.register() checks if email exists
    ↓
Password encrypted with BCrypt
    ↓
User saved to MySQL users table
    ↓
Redirect to /login with success message
```

### Login Flow:
```
Browser fills login form
    ↓
POST /login with email + password
    ↓
Spring Security intercepts, calls CustomUserDetailsService
    ↓
Looks up user by email in database
    ↓
Compares password with stored hash
    ↓
If correct: creates session, redirects to /dashboard
    ↓
Also: JwtTokenProvider creates a JWT token
```

### File Upload Flow:
```
Browser selects a file on /upload page
    ↓
POST /upload with file
    ↓
FileStorageService validates (size, type)
    ↓
Saves to uploads/ folder with UUID name
    ↓
OcrService extracts raw text (placeholder)
    ↓
GeminiService sends file to AI for analysis
    ↓
AI returns extracted fields (name, dates, etc.)
    ↓
Document saved to documents table in MySQL
    ↓
Redirect to /dashboard showing new document
```

### Admin Deletion Flow:
```
Admin clicks "Delete" on a user
    ↓
POST /admin/users/{id}/delete
    ↓
User + all their documents fetched from DB
    ↓
userRepository.delete(user) — cascades to documents
    ↓
FileStorageService deletes actual files from uploads/
    ↓
Redirect to /admin with success message
```
